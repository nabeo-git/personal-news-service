# ai-news-service PLAN.md

## このドキュメントの目的
技術選定・実装手順ではなく、**プロジェクトをどう進めるか**の進め方と考え方を定義する。
技術選定は後続フェーズで比較検討の上で決定する。

---

## プロジェクト概要
AI駆動のパーソナライズニュースレターサービス。  
仕様の詳細: `初期構築仕様書.txt`

### 実現したいこと（要約）
1. **ニュース収集** — 関心カテゴリに沿ったパーソナライズフィード。不要記事をAIが学習・除外
2. **金融インサイト** — 株価動向をAIが分析・記事生成して配信
3. **タスク通知** — Gmail・Todoアプリをスキャンし、期限・要返信メールを通知

---

## 進め方（フェーズ構成）

### Phase 0: 要件定義の精査
**目的:** 仕様書の曖昧な部分を潰し、スコープを確定させる  
**成果物:** 確定要件リスト + 未検討事項の洗い出し

- 仕様書の3機能を1つずつレビュー
- 不明点・前提条件・制約の整理
- MVP（最小実装）のスコープを合意
- 除外スコープを明示（後回しにするものを明確化）

---

### Phase 1: 技術調査・選定
**目的:** 要件に対して選択肢を比較し、根拠をもって技術を決定する  
**成果物:** 技術選定ドキュメント（各選択肢の比較表 + 採用理由）

調査対象の例（確定ではない）:
- バックエンド言語・フレームワーク（TypeScript/Python/その他）
- データベース（RDB / NoSQL / ベクトルDB）
- LLM・AI基盤（Claude API / Gemini / OpenAI など）
- ニュースデータ取得方法（スクレイピング / ニュースAPI / RSS）
- ワークフロー自動化（n8n / Zapier / cron + Lambda / その他）
- 認証・個人データアクセス（OAuth 実装方法）
- デプロイ先（AWS / GCP / Fly.io / Vercel / その他）

> 各項目で「なぜそれを選ぶか」「他の選択肢との差異」を明示することが必須。

---

### Phase 2: アーキテクチャ設計
**目的:** 決定した技術を元にシステム構成を設計する  
**成果物:** アーキテクチャ図 + コンポーネント一覧 + データフロー

- コンポーネント分割（サービス境界の決定）
- データモデル設計（スキーマ）
- 3機能の統合ポイントの設計（共有するもの / 独立させるもの）
- セキュリティ・認証フローの設計

---

### Phase 3: GitHubリポジトリ初期化
**目的:** 開発環境を整備し、実装をスタートできる状態にする  
**成果物:** GitHubリポジトリ + 初期ディレクトリ構造

- リポジトリ作成（Private）
- モノレポ or マルチレポの決定（Phase 1で検討）
- 基本ファイル: CLAUDE.md, .env.example, README.md
- CI/CD パイプラインの基本設定

---

### Phase 4以降: 実装
Phase 0〜3の結果を踏まえて計画する。現時点では定義しない。
