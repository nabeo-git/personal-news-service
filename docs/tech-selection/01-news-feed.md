# F-01 ニュースフィード 技術選定

## このドキュメントの目的

技術選定の**結論だけでなく、なぜその選択をしたか・何を捨てたか**を記録する。
将来「なぜこの技術にしたのか」を見直す際の根拠として使う。

## 選定対象

F-01の実装に必要な技術要素を5つに分解して選定する。

| # | 技術要素 | 選定結果 |
|---|---------|---------|
| 1 | ニュース取得方法 | **RSS + GNews API（無料枠）** |
| 2 | LLM | **Claude API（Haiku）** |

## 1. ニュース取得方法

### 要件の整理

- 特定テーマ（例: 「ChatGPT 新機能」）でニュースを検索できること
- 無料 or 低コストで運用できること
- 個人利用のため1日あたりのリクエスト数は少ない（1回/日 × カテゴリ数）
- 記事の全文は必須ではない（タイトル・概要・URLがあればLLMで判断できる）

### 検討した選択肢

#### 選択肢A: RSS フィード

**概要**: 各メディアのRSSを購読して取得する。

**メリット**:
- 完全無料
- リクエスト制限なし
- ソースを自分でコントロールできる

**デメリット**:
- カバーするメディアを自分でリスト管理する必要がある
- テーマ横断の「横串検索」ができない（Google News RSSは例外）
- メディアによってRSSの質が異なる（更新が遅いものも）

**補足**: Google News の検索URL (`https://news.google.com/rss/search?q=...`) をRSSとして利用する方法がある。これによりテーマ検索が可能になる。

#### 選択肢B: NewsAPI.org

**概要**: 有名なニュースAPIサービス。

**メリット**:
- 検索精度が高い
- ドキュメントが充実

**デメリット**:
- **無料プラン（Developer）は記事が24時間遅延する** — ニュースの鮮度が失われる
- 有料プランは$449/月〜 — 個人利用に対してコストが高すぎる
- **→ 却下**: 無料では鮮度要件を満たせない

#### 選択肢C: GNews API

**概要**: Google Newsをベースにした軽量なニュースAPI。

**メリット**:
- 無料枠: 100リクエスト/日（個人利用には十分）
- 有料プランが$9/月〜と安価
- テーマ・言語・国での絞り込みが可能
- 記事はリアルタイムに近い鮮度

**デメリット**:
- ソースの制御が難しい（どのメディアが含まれるか不透明）
- 日本語ニュースのカバレッジは英語より弱い

#### 選択肢D: Webスクレイピング（ScrapeGraphAI等）

**メリット**:
- 特定サイトを直接取得でき、精度が高い

**デメリット**:
- サイトの構造変更に弱い、メンテコストが高い
- robots.txtの遵守など法的考慮が必要
- MVPとしては複雑すぎる
- **→ 却下**: MVP段階では採用しない

### 決定: Google News RSS + GNews API（無料枠）の併用

**理由**:

1. **Google News RSS（メイン）**: `https://news.google.com/rss/search?q=<query>&hl=ja` をカテゴリごとに取得する。無料・リアルタイム・テーマ検索が可能で、個人利用のほとんどのニーズを満たす。

2. **GNews API（補助）**: Google News RSSでカバーできないテーマや、英語の技術ニュース収集に利用する。無料枠100回/日は1日1回の実行なら十分。

**捨てたこと**: スクレイピングによる細かいソース制御は、コストとリスクに見合わないためMVPでは見送る。

## 2. LLM

### 要件の整理

- 収集した記事（タイトル・概要）を見て「除外条件に合致するか」を判断する
- 記事の日本語要約を生成する
- 1日1回の実行で、処理する記事数はカテゴリ × 件数（例: 5カテゴリ × 20件 = 100件/日）
- コストを抑えたい

### 検討した選択肢

#### LLM コスト・性能比較（2025年時点）

| モデル | Input | Output | 特徴 |
|--------|-------|--------|------|
| Claude Haiku 4.5 | $0.80/M tokens | $4/M tokens | 高速・安価・日本語◎ |
| Claude Sonnet 4.6 | $3/M tokens | $15/M tokens | 高精度・中程度のコスト |
| GPT-4o mini | $0.15/M tokens | $0.60/M tokens | 最安値レベル |
| Gemini 1.5 Flash | $0.075/M tokens | $0.30/M tokens | 最安値・大きなコンテキスト窓 |

#### コスト試算（100記事/日、記事1件あたり平均200tokens想定）

| モデル | 月間インプットコスト概算 |
|--------|----------------------|
| Claude Haiku | 100件 × 200tokens × 30日 = 600K tokens → **$0.48/月** |
| GPT-4o mini | 同 → **$0.09/月** |
| Gemini Flash | 同 → **$0.045/月** |

**→ いずれも月額1ドル以下。コストは差別化要因にならない。**

#### 選択肢A: Claude API（Haiku）

**メリット**:
- 日本語の品質が高い（ニュース要約・除外判断の精度が高い）
- このプロジェクト自体がClaude Code環境で開発しており、APIキーの管理が一元化できる
- Tool use / structured output が扱いやすい

#### 選択肢B: GPT-4o mini / Gemini Flash

**メリット**: さらに安い

**デメリット**:
- 別途APIキーの管理が必要
- コスト差が月数十円レベルでは差別化要因にならない

### 決定: Claude API（Haiku）

**理由**:

- コストは月額100円以下でいずれのモデルも大差なし
- 日本語ニュースを扱うため、日本語品質が高いClaudeを選択
- 開発環境と同じエコシステムで管理できる（APIキー1種類）
- 将来的に精度を上げたい場合はSonnetに切り替えるだけで対応できる

**使い分けの方針**:
- 除外判断（分類タスク）→ Haiku（安価・高速で十分）
- 最終的なニュースレター文章生成 → Haiku（ただし品質が不満なら Sonnet に切り替え）

## 3. 配信手段

### 要件の整理

- 毎朝、フォーマットされたニュースを受け取れる
- セットアップが簡単（OAuthやドメイン設定などの複雑な手順を避けたい）
- Markdown or テキストで読みやすく届く

### 検討した選択肢

#### 選択肢A: Slack Webhook

**概要**: Incoming Webhookを作成し、HTTPのPOST1回で送信できる。

**メリット**:
- セットアップ: Slack App設定でWebhook URLを発行するだけ（5分以内）
- 認証: Webhook URLを環境変数に入れるだけ（OAuthや送信ドメイン設定不要）
- Markdownがほぼそのままレンダリングされる（見やすい）
- スレッド機能でカテゴリごとに分けて送信できる
- スマートフォンからも確認しやすい

**デメリット**:
- Slackを使っていない環境では使えない
- 無料プランでは90日分のメッセージ履歴しか残らない

#### 選択肢B: メール（SendGrid 無料枠）

**概要**: SendGridのAPIでメールを送信する。

**メリット**:
- 普遍的（誰でも受け取れる）
- 履歴がGmailに残る

**デメリット**:
- 送信ドメインの設定（SPF/DKIM）が必要
- HTMLメールの場合はテンプレート設計が必要
- セットアップがSlackより手間

#### 選択肢C: Gmail API（自分のGmailから自分に送信）

**メリット**: 既存のGmailに届く

**デメリット**:
- OAuthの設定が必要（認証フローが複雑）
- F-03でGmail連携を予定しており、その前にOAuth実装するのは手間が重複する
- **→ F-03実装時にまとめて検討する**

### 決定: Slack Webhook

**理由**:

- MVPとして最小のセットアップで動かすことを優先
- Webhook URLを1つ環境変数に入れるだけで完結する
- MarkdownがそのままSlackで見やすく表示される
- 毎朝スマートフォンで確認する運用に向いている

**捨てたこと**: メールへの対応はF-01 MVP後に必要であれば追加する。OAuthを伴うGmail連携はF-03まで持ち越す。

## 4. スケジューラー

### 要件の整理

- 毎朝定時（例: 7:00 JST）に自動実行される
- 管理コストが低い（サーバーを常時稼働させたくない）
- 実行ログが確認できる

### 検討した選択肢

#### 選択肢A: GitHub Actions（cron schedule）

**概要**: `.github/workflows/` にcron設定を書くだけで定時実行できる。

**メリット**:
- **インフラゼロ**: サーバー不要・コスト不要（private repoで月2000分の無料枠あり）
- ログがGitHub上で確認できる
- 実行環境がコードと同じリポジトリで管理される
- 失敗時のメール通知が標準搭載

**デメリット**:
- GitHub Actionsのcronは**最大15分程度の遅延**が発生することがある（正確な時刻に動かしたい場合は不向き）
- リポジトリがprivateの場合、月2000分の制限あり（1日1回 × 5分 = 月150分 → 十分）

#### 選択肢B: AWS Lambda + EventBridge

**メリット**:
- 正確な時刻実行
- 月100万回まで無料

**デメリット**:
- AWS設定（IAM, Lambda, EventBridge）が必要でセットアップコストが高い
- デプロイフローの整備が必要

#### 選択肢C: VPS上でcron

**メリット**: シンプル、正確

**デメリット**:
- 常時稼働のサーバーが必要（月額コスト）
- サーバー管理が発生する

#### 選択肢D: n8n（セルフホスト or クラウド）

**メリット**: GUIでワークフローが組める、将来的な拡張性が高い

**デメリット**:
- セットアップコストが高い
- F-01 MVPとしてはオーバースペック

### 決定: GitHub Actions

**理由**:

- インフラゼロで始められる点がMVPの原則に合致する
- ログ・デプロイ・スケジューラーがすべてGitHubに集約されてシンプル
- 実行頻度（1日1回）ならリソース制限に引っかからない
- 15分の遅延は「毎朝7時に届く」用途では許容範囲

**将来の移行判断**: 正確な時刻が必要になった場合や、処理が重くなった場合にLambda or VPSに移行する。

## 5. 除外条件の保存形式

### 要件の整理

- ユーザーが「不要」と判断した記事から除外条件を保存する
- 次回実行時に条件を読み込んで、合致する記事をフィルタリングする
- 除外条件を手動で確認・編集できる
- バージョン管理できると望ましい（いつ何を除外したかのの履歴）

### 検討した選択肢

#### 選択肢A: JSONファイル（リポジトリ管理）

**概要**: `data/exclusion-rules.json` として保存し、Gitで管理する。

**メリット**:
- インフラ不要、追加コストゼロ
- テキストエディタで直接編集できる（透明性が高い）
- Gitの履歴で変更追跡できる
- 構造がシンプルで理解しやすい

**デメリット**:
- GitHub ActionsからGitリポジトリに書き込む場合、コミット権限の設定が必要
- 同時書き込みの競合が起きうる（個人利用なのでほぼ問題ない）

**補足（書き込みの方法）**:
- GitHub Actions実行中にJSONを更新してコミット・プッシュする方法（`actions/checkout` + `git commit` ステップ）
- または外部ストレージ（後述）に書き込む方法

#### 選択肢B: SQLite

**メリット**: クエリが使える、履歴管理が楽

**デメリット**:
- GitHub Actions環境では永続化できない（実行ごとにリセットされる）
- 別途S3等のストレージが必要になる → コスト・複雑さが増す

#### 選択肢C: ベクトルDB（Pinecone等）

**メリット**: 「意味的に近い記事」を高精度で除外できる

**デメリット**:
- MVP段階ではオーバースペック
- コスト・複雑さが増す
- LLMによる判断（「この記事は過去の除外条件に近いか？」をLLMに聞く）で代替できる

### 決定: JSONファイル（リポジトリ管理）+ LLMによる類似判断

**理由**:

- 除外条件はシンプルなテキストのリストで十分。ベクトルDBは不要
- 「ベクトル類似度計算」の代わりに **LLMに「この記事は以下の除外条件に該当しますか？」と聞く** ことで、意味的な判断が可能
- 設定ファイルとしてリポジトリで管理することで、変更履歴が残り、手動編集も容易

**JSONの構造（案）**:

```json
{
  "exclusion_rules": [
    {
      "id": "rule_001",
      "description": "中国の規制・政策ニュース",
      "keywords": ["中国", "規制", "政策"],
      "added_at": "2026-05-02T07:00:00Z",
      "source_article_title": "中国がAI規制を強化..."
    }
  ]
}
```

**捨てたこと**: ベクトルDBによる高精度マッチングはF-01 MVP後の品質改善フェーズで検討する。

## 決定済み技術スタック（F-01 MVP）

| 技術要素 | 採用技術 | 補足 |
|---------|---------|------|
| ニュース取得 | Google News RSS + GNews API（無料枠） | Python `feedparser` ライブラリを使用予定 |
| LLM | Claude API（Haiku） | 除外判断・要約の両方に使用 |
| 配信 | Slack Webhook | Incoming Webhook URL を環境変数で管理 |
| スケジューラー | GitHub Actions（cron） | `.github/workflows/daily-news.yml` |
| 除外条件保存 | JSON（`data/exclusion-rules.json`） | Gitリポジトリで管理 |
| 実装言語 | Python | RSS解析・API呼び出しのライブラリが充実 |

### 主要なPythonライブラリ（候補）

| ライブラリ | 用途 |
|-----------|------|
| `feedparser` | RSS解析 |
| `httpx` | HTTP通信（News API呼び出し、Slack通知） |
| `anthropic` | Claude API呼び出し |
| `pydantic` | 設定・データ構造のバリデーション |
